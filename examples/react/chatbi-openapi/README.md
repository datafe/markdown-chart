# React + ChatBI OpenAPI example

This example shows a third-party React application rendering the Markdown
stream returned by ChatBI. The browser uses `MarkdownChart` in simple mode and
delegates both data access and temporary chart-source conversion to the host:

```tsx
const resolveDataRef = createChatBIArtifactDataRefResolver({
  sessionId,
  requestId,
});
const resolver = createChatBILegacyResolver({
  resolveDataRef,
  executeChartSource: createExecuteChartSource(),
});
const echarts = useMemo(() => ({ resolveDataRef }), [resolveDataRef]);

<MarkdownChart
  source={markdown}
  streaming={streaming}
  echarts={echarts}
  resolveLegacyEChartQuery={resolver}
/>
```

Memoize the `echarts` options object as shown. A stable identity lets completed
chart blocks stay mounted while more Markdown tokens arrive, instead of
recreating the registry and repeating artifact requests.

Pass the current `requestId` whenever it is available. It limits artifact
lookup to the current ChatBI request. Without it, the list operation can only
search the entire session, so an old artifact with the same name can make the
result ambiguous.

## Browser-to-backend contracts

The example uses three same-origin endpoints owned by the third-party
application:

| Browser endpoint | Backend responsibility |
| --- | --- |
| `POST /api/dataworks/list-agent-session-artifacts` | Sign and call [`ListAgentSessionArtifacts`](https://help.aliyun.com/zh/dataworks/developer-reference/api-dataworks-public-2024-05-18-listagentsessionartifacts). Forward the JSON-RPC response. |
| `POST /api/dataworks/get-agent-session-artifact-meta` | Sign and call [`GetAgentSessionArtifactMeta`](https://help.aliyun.com/zh/dataworks/developer-reference/api-dataworks-public-2024-05-18-getagentsessionartifactmeta). Forward the JSON-RPC response. |
| `POST /api/chart/execute` | Execute the temporary chart source in the host's isolated, interruptible sandbox and return JSON only. |

The browser sends the official JSON-RPC request shape to the first two proxy
routes. The list resolver includes `SessionId`, optional `RequestId`,
`MaxResults`, and `NextToken`; it follows every page and requires exactly one
`ArtifactName` equal to the requested `${jobId}.csv`. It then calls the meta
route with `SessionId + ArtifactPath`, reads `ArtifactContent`, and parses the
CSV into a renderer-neutral dataset.

The third route receives:

```json
{
  "source": "the chart source from the Markdown block",
  "inputData": [{ "category": "A", "value": 10 }]
}
```

It must return a JSON object shaped as `{ "spec": { ... } }`. The sample
browser never uses `eval` or `new Function`; the resulting spec is validated
again by `markdown-chart` before ECharts loads.

Implement the DataWorks proxy with the official Alibaba Cloud SDK or the
signed server-side code generated by OpenAPI Explorer. AccessKey credentials,
temporary signing credentials, and signing logic must never be sent to the
browser. Apply your application's authorization checks to all three routes and
honor client disconnect/abort signals in the sandbox service. The proxy must
also enforce its own response/file-size limit before it buffers, reads, or
returns `ArtifactContent`; the browser resolver's response, content, row,
column, and cell limits are only a second line of defense. You can override the
browser defaults with the resolver's `limits` option when the host needs a
stricter policy.

Run the example after providing those backend routes:

```sh
pnpm --filter @datafe/markdown-chart-example-react-chatbi-openapi dev
```
